A1 · FundamentalsCore Java

Classes & Objects

A class is a blueprint: it declares what data an entity holds (fields) and what it can do (methods). An object is one concrete thing built from that blueprint, living in memory with its own field values. You write the class once; you create many objects from it.

ETL analogy

A class is like a table definition / mapping schema in Informatica — the column list, the contract. An object is a single row flowing through the pipeline: same shape as the schema, but its own values. A method on the class is a reusable transformation you can apply to any row.

The four pillars interviewers expect you to name: Encapsulation (hide fields behind methods), Inheritance (a class reuses/extends another), Polymorphism (one reference type, many runtime behaviours), and Abstraction (program to an interface, not an implementation).

Encapsulated class + object creation
public class Account {
    private String owner;        // private = encapsulated state
    private double balance;

    public Account(String owner, double opening) {  // constructor
        this.owner = owner;
        this.balance = opening;
    }

    public void deposit(double amt) {               // behaviour
        if (amt <= 0) throw new IllegalArgumentException("amount must be > 0");
        this.balance += amt;
    }

    public double getBalance() { return balance; }  // controlled read access
}

// --- somewhere else ---
Account a = new Account("Nag", 1000);   // 'a' is an OBJECT of class Account
a.deposit(500);
System.out.println(a.getBalance());     // 1500.0

Interfaces and inheritance let one reference point at many implementations — the basis of polymorphism, and of how Spring swaps implementations for you:

interface Notifier { void send(String msg); }

class EmailNotifier implements Notifier {
    public void send(String msg) { System.out.println("EMAIL: " + msg); }
}
class SmsNotifier implements Notifier {
    public void send(String msg) { System.out.println("SMS: " + msg); }
}

Notifier n = new EmailNotifier();   // declared type Notifier, runtime type EmailNotifier
n.send("Batch finished");           // calls EmailNotifier.send  -> polymorphism
Gotchas

== vs equals(). == compares references (same object in memory); .equals() compares logical value. For your own classes, if you override equals() you must also override hashCode() (this comes back in A3). Static vs instance: a static member belongs to the class, shared by all objects; an instance member belongs to each object. Null: calling a method on a null reference throws NullPointerException — the single most common Java bug.

Likely interview questions
What's the difference between a class and an object?

A class is the compile-time blueprint (fields + methods); an object is a runtime instance created with new, holding its own state on the heap. One class, many objects.

Explain the four OOP principles with one line each.

Encapsulation — keep fields private, expose behaviour via methods. Inheritance — extends to reuse a parent's contract. Polymorphism — same call resolves to the runtime type's method. Abstraction — depend on interfaces so implementations can change.

Difference between == and .equals()?

== tests reference identity; .equals() tests value equality (as defined by the class). String, the wrapper types, and records implement value-based equals().

What is a constructor and can you have more than one?

A constructor initialises a new object; it has the class's name and no return type. Yes — you can overload constructors with different parameter lists. If you write none, the compiler supplies a no-arg default.

Hands-on

Create a BatchJobRun class with fields jobName, rowsRead, rowsWritten and a method skipped() returning the difference. Make a List<BatchJobRun> of three runs and print the one with the most skipped rows. (You'll redo this with Streams in P1 — keep it.)

You've used these without the Java names. A HashMap is a lookup/cache transform — key in, row out, no order. A TreeMap is a sorted aggregator. A HashSet is a distinct / dedup step. An ArrayList is just an ordered staging buffer. Choosing the right one is the same instinct as choosing the right transform.

List<String> staging   = new ArrayList<>();           // ordered, duplicates ok

counts.merge("AAPL", 1, Integer::sum);  // idiomatic "increment or insert"

The "Big-O" you actually need to recite: array index = O(1); HashMap get/put = O(1) average; TreeMap = O(log n); searching an unsorted list = O(n). That's enough for most enterprise interviews.

Gotchas

ArrayList vs LinkedList: in practice ArrayList wins almost always (cache-friendly); only reach for LinkedList when you constantly add/remove at the head. Don't use Vector/Stack (legacy, synchronised); use ArrayList/ArrayDeque. Modifying a collection while iterating it with a for-each throws ConcurrentModificationException — use an Iterator.remove() or removeIf().

Likely interview questions
List vs Set vs Map — when do you use each?

List = ordered, indexed, duplicates allowed. Set = unique elements, membership tests. Map = key→value lookups. Pick by the question you'll ask the data most often.

ArrayList vs LinkedList?

ArrayList = O(1) random access, contiguous memory, resizes by copying. LinkedList = O(1) insert/remove at ends but O(n) random access and poor cache locality. Default to ArrayList.

HashMap vs TreeMap vs LinkedHashMap?

HashMap — fastest, no order. TreeMap — sorted by key, O(log n), supports range queries. LinkedHashMap — keeps insertion (or access) order, great for LRU caches.

How would you remove duplicates from a list?

new ArrayList<>(new LinkedHashSet<>(list)) to keep order, or list.stream().distinct().toList().

Hands-on

Given a list of stock trades (symbol, qty), build a Map<String,Integer> of total quantity per symbol using merge(), then print it sorted by symbol (hint: wrap in a TreeMap). This is a daily-aggregation pattern you'll recognise from ETL.

A3 · InternalsHigh-frequency question

How a HashMap Works

This is one of the most asked Java questions. A HashMap stores key→value pairs in an array of buckets. To find a key it does three things: compute the key's hashCode(), spread those bits and map them to a bucket index, then walk that bucket comparing keys with equals(). Because a good hash spreads keys evenly, each bucket usually holds one entry — so lookup is O(1) on average.

ETL analogy

A HashMap is a hash-join / lookup cache. The "build" side hashes keys into buckets; the "probe" side hashes the incoming key to jump straight to the right bucket instead of scanning every row. Collisions in a HashMap are the same concept as hash-bucket skew in a join.

put / get and the contract
Map<String,Integer> m = new HashMap<>();
m.put("AAPL", 189);          // hash("AAPL") -> bucket; store entry
m.put("AAPL", 190);          // same bucket, equals() matches -> value replaced
Integer v = m.get("AAPL");   // 190, O(1) average

m.getOrDefault("TSLA", 0);   // 0 if absent
m.computeIfAbsent("watch", k -> new ArrayList<>()).add("AAPL"); // multimap pattern

The critical rule: keys must implement hashCode() and equals() consistently. If two objects are equals(), they must return the same hashCode(). Break this and the map "loses" your keys — you put a value in and can never get it back, because lookup hashes to the wrong bucket.

Gotchas

Mutable keys are a trap: if you mutate a field used in hashCode() after inserting, the entry becomes unreachable. Prefer immutable keys (String, wrapper types, or records — records auto-generate correct equals/hashCode). HashMap is not thread-safe — concurrent writes can corrupt it or loop forever on old JDKs; use ConcurrentHashMap for shared mutable maps. Initial capacity: default 16, load factor 0.75 → it resizes (rehashes everything) when 75% full; pre-size large maps with new HashMap<>(expected).

Likely interview questions
Walk me through what happens on map.put(key, value).

Compute key.hashCode(), spread its bits, derive a bucket index with hash & (n-1). Go to that bucket. If empty, insert. If occupied, walk the chain comparing with equals(); if a key matches, replace its value, else append. Resize/rehash if the table passes the load-factor threshold.

Why must equals() and hashCode() agree?

Lookup uses hashCode() to find the bucket and equals() to find the entry within it. If equal objects had different hashes, get() would search the wrong bucket and never find the entry. The contract keeps the two steps consistent.

What changed in Java 8's HashMap?

Buckets that grow beyond a threshold (8 entries, table size ≥ 64) convert from a linked list to a red-black tree, improving the worst case from O(n) to O(log n) under heavy collisions or hash-flooding attacks.

HashMap vs ConcurrentHashMap?

HashMap is unsynchronised — undefined behaviour under concurrent modification. ConcurrentHashMap allows concurrent reads and segmented/locked writes (CAS + bin-level locks), with no need to lock the whole map.

Hands-on

Write a record CcyPair(String base, String quote) and use it as a HashMap key for FX rates. Put a few rates, then prove lookup works because the record auto-generates equals/hashCode. Then write a normal class with the same fields but no overrides and watch get() return null — that failure is the whole lesson.

A4 · Java 8Core Javaenables lambdas

Functional Interfaces

A functional interface is simply an interface with exactly one abstract method (a "SAM" type — Single Abstract Method). That single method is what a lambda or method reference provides the body for. This is the foundation everything in Streams (P1) is built on.

Your own functional interface
@FunctionalInterface              // optional, but compiler enforces "one abstract method"
interface RowTransform {
    String apply(String row);     // the single abstract method
}

RowTransform upper = row -> row.toUpperCase();      // lambda implements it
RowTransform trim  = String::trim;                  // method reference

System.out.println(upper.apply("aapl"));  // AAPL

You rarely need to write your own — java.util.function ships the ones you'll use constantly:

InterfaceAbstract methodMeansStream use
Function<T,R>R apply(T)transform T → R.map()
Predicate<T>boolean test(T)true/false test.filter()
Consumer<T>void accept(T)do something, no return.forEach()
Supplier<T>T get()produce a valueorElseGet()
BiFunction<T,U,R>R apply(T,U)two inputs → one outreduce()
UnaryOperator<T>T apply(T)T → Tin-place map
ETL analogy

These map one-to-one onto pipeline steps you already build: Function = a transformation expression, Predicate = a filter condition, Consumer = a target writer, Supplier = a source/sequence generator. A lambda is just an inline transform with no class boilerplate.

The built-ins in action
import java.util.function.*;

Predicate<Integer> big   = n -> n > 100_000;
Function<String,Integer> len = String::length;
Consumer<String> log     = System.out::println;
Supplier<Double> rnd     = Math::random;

System.out.println(big.test(150_000));   // true
System.out.println(len.apply("AAPL"));   // 4
log.accept("writing row");               // writing row

// composition: predicates and functions chain
Predicate<Integer> bigAndEven = big.and(n -> n % 2 == 0);
Function<String,Integer> lenPlus1 = len.andThen(x -> x + 1);
Gotchas

A lambda can only capture effectively final local variables (you can't reassign a captured local). this inside a lambda refers to the enclosing object, not the lambda (unlike an anonymous class). Don't overuse method references when a lambda reads clearer. And an interface with two abstract methods is not functional — @FunctionalInterface makes the compiler tell you.

Likely interview questions
What is a functional interface? Give examples.

An interface with exactly one abstract method, usable as the target type of a lambda or method reference. Examples: Runnable, Comparator, Callable, and the java.util.function family (Function, Predicate, Consumer, Supplier).

Can a functional interface have default or static methods?

Yes — only the count of abstract methods must be one. default and static methods are allowed (that's how Predicate.and()/Function.andThen() exist).

Lambda vs anonymous inner class?

Lambdas are concise, don't introduce a new scope for this, and don't generate a separate .class file per use (invoked via invokedynamic). Anonymous classes can have state and implement multi-method interfaces.

What are the four kinds of method reference?

Static (Integer::parseInt), bound instance (System.out::println), unbound instance (String::toUpperCase), and constructor (ArrayList::new).

Hands-on

Declare a Predicate<Trade> for "qty > 1000", a Function<Trade,String> for "symbol uppercased", and a Consumer<Trade> that prints it. Combine the predicate with .and() to also require a specific symbol, and apply all three over a list with a plain loop. Next module shows the Stream version.

A5 · WebSpring Bootdeep-dived in P4

REST API Call

REST is a style for exposing data over HTTP. A client sends a request (method + URL + optional body) and the server returns a response (status code + body, usually JSON). The HTTP method states intent: GET read, POST create, PUT replace, PATCH partial update, DELETE remove.

ETL analogy

A REST endpoint is a real-time, on-demand pipeline: instead of a scheduled batch reading a file, a single request is the "source", your service logic is the "transform", and the JSON response is the "target". A GET is a query; a POST is an insert.

Server side — exposing an endpoint in Spring Boot is a few annotations:

CustomerController.java — the server
@RestController
@RequestMapping("/api/customers")
public class CustomerController {

    private final CustomerService service;
    public CustomerController(CustomerService service) { this.service = service; }

    @GetMapping("/{id}")                         // GET /api/customers/42
    public Customer one(@PathVariable Long id) {
        return service.findById(id);
    }

    @GetMapping                                  // GET /api/customers?active=true
    public List<Customer> list(@RequestParam(defaultValue = "true") boolean active) {
        return service.findAll(active);
    }

    @PostMapping                                 // POST /api/customers  (JSON body)
    @ResponseStatus(HttpStatus.CREATED)          // 201
    public Customer create(@RequestBody @Valid Customer c) {
        return service.save(c);
    }
}

Client side — calling someone else's API. The modern JDK has a built-in HttpClient (no library needed); inside Spring you'd use RestClient (Boot 3.2+).

Plain JDK 21 HttpClient — the client
import java.net.http.*;
import java.net.URI;

HttpClient client = HttpClient.newHttpClient();
HttpRequest req = HttpRequest.newBuilder()
        .uri(URI.create("https://api.example.com/api/customers/42"))
        .header("Accept", "application/json")
        .GET()
        .build();

HttpResponse<String> res = client.send(req, HttpResponse.BodyHandlers.ofString());
System.out.println(res.statusCode());   // 200
System.out.println(res.body());         // {"id":42,...}
Spring RestClient — the idiomatic in-app client
RestClient rest = RestClient.create();
Customer c = rest.get()
        .uri("https://api.example.com/api/customers/{id}", 42)
        .retrieve()
        .body(Customer.class);   // JSON auto-deserialised to your record/class

Status codes you must know: 2xx success (200 OK, 201 Created, 204 No Content), 4xx client error (400 Bad Request, 401 Unauthorised, 403 Forbidden, 404 Not Found), 5xx server error (500, 503).

Gotchas

REST is stateless — never store per-client session state in the controller; pass what's needed each call (this is why JWTs exist, see P4). Use the right method: a GET must not change data (it should be safe and idempotent). Return proper status codes, not always 200 with an error in the body. @RequestBody needs JSON + the right Content-Type: application/json header or you'll get 415.

Likely interview questions
What makes an API "RESTful"?

Resources addressed by URLs, manipulated with standard HTTP methods, stateless requests, and a uniform interface (typically JSON). Each request carries everything the server needs.

PUT vs POST vs PATCH?

POST creates (not idempotent). PUT replaces a resource wholesale (idempotent). PATCH partially updates. Repeating a PUT leaves the same state; repeating a POST may create duplicates.

What's idempotency and which methods are idempotent?

Same request repeated yields the same server state. GET, PUT, DELETE are idempotent; POST generally is not. It matters for retries.

How do you handle errors in a Spring REST API?

Throw domain exceptions and translate them centrally with @RestControllerAdvice + @ExceptionHandler, returning the right status and a consistent error body. Covered in P4.

Hands-on

Once you've done P4 setup, expose GET /api/ping returning {"status":"ok"} and call it from a second tiny Java main using HttpClient. Then add POST /api/echo that returns whatever JSON you send it. You'll have built and consumed a REST API end to end.

A6 · Modern JavaJDK 21 LTS

Java 21 Features — the headline answer

Java 21 (Sept 2023) is the LTS the role targets. If asked "what's new in Java 21," lead with virtual threads, then pattern matching for switch + record patterns, then sequenced collections, and mention structured concurrency as a preview. Records, sealed classes, text blocks and switch expressions arrived earlier (Java 14–17) but are part of the same "modern Java" story. Each is detailed in Part B; here's the interview-ready summary.

FeatureSinceOne-line pitch
Virtual threads21 (final)Millions of cheap threads; write blocking code that scales like async.
Pattern matching for switch21 (final)Switch on type, with guards — no cast boilerplate.
Record patterns21 (final)Destructure records directly in switch/instanceof.
Sequenced collections21 (final)getFirst()/getLast()/reversed() across List/Set/Map.
Records16Immutable data carriers, zero boilerplate.
Sealed classes17Restrict who can implement — exhaustive modelling.
Text blocks15Multi-line strings (SQL/JSON) without escaping.
Switch expressions14switch that returns a value, arrow syntax.
Structured concurrency21 (preview)Treat a group of subtasks as one unit of work.
ETL analogy

Virtual threads are the one to internalise: imagine running thousands of parallel source-reads or API lookups without exhausting a thread pool — like scaling pipeline parallelism without scaling hardware. Record patterns are like decomposing a structured row into its fields in a single expression.

A taste of three Java 21 features together
sealed interface Event permits Trade, Heartbeat {}
record Trade(String symbol, int qty) implements Event {}
record Heartbeat(long ts) implements Event {}

String describe(Event e) {
    return switch (e) {                                  // pattern matching for switch
        case Trade(String sym, int q) when q > 1000 ->   // record pattern + guard
                "BIG " + sym + " x" + q;
        case Trade(String sym, int q) -> sym + " x" + q;
        case Heartbeat h            -> "ping@" + h.ts();
    };                                                   // exhaustive: no default needed
}

// Virtual threads: 10,000 concurrent tasks, trivially
try (var ex = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10_000; i++) ex.submit(() -> fetchQuote());
}
Gotchas

Because the type hierarchy is sealed, the switch is exhaustive and needs no default — add a new permitted type and the compiler forces you to handle it. Structured concurrency is still a preview in 21 (needs --enable-preview); know it conceptually, don't claim production use. Note Java 25 is now the newer LTS (Sept 2025) — fine to mention, but 21 is what this role asks for.

Likely interview questions
What's the single biggest feature in Java 21?

Virtual threads (Project Loom, JEP 444). They make blocking I/O cheap to scale: you keep simple thread-per-request code but can run millions of them, because a blocked virtual thread unmounts from its carrier OS thread. See P3.

What are record patterns?

They let you destructure a record's components inside instanceof or switch — e.g. case Trade(String sym, int q) binds the fields directly, no manual accessors or casts.

What are sequenced collections?

New interfaces (SequencedCollection/SequencedMap) giving a uniform getFirst(), getLast(), addFirst(), reversed() across ordered collections — previously inconsistent across List/Deque/LinkedHashMap.

Hands-on

Build the Event hierarchy above and a List<Event> of mixed trades and heartbeats; map each to its description with the switch. Then try removing one case and watch the compiler reject it — that's exhaustiveness from sealed types doing its job.

A7 · DataSQL · your home turf

SQL Indexes & Fast Retrievals

This is the topic closest to your 20 years of experience — frame it that way. An index is a separate, sorted data structure (almost always a B-tree) that lets the database find rows without scanning the whole table. Without an index, finding one row in a million is a full table scan — O(n). With a B-tree index it's roughly O(log n): a handful of page reads.

ETL analogy

You already tune this. An index is the database's version of a pre-sorted lookup / hash cache on a join key. Choosing index columns is the same decision as choosing partition/sort keys so your loads and lookups don't scan everything. "Methods to get fast retrievals" = the same toolkit you use to make ETL targets queryable.

Index types and when to use them

TypeBest for
B-tree (default)Equality and range: =, <, >, BETWEEN, ORDER BY, prefix LIKE 'abc%'
Composite (multi-column)Filtering on several columns together — order matters (leftmost-prefix rule)
Covering (INCLUDE)Index holds all selected columns → answer from index, never touch the table
HashPure equality only (no ranges); niche in Postgres
PartialIndex only rows matching a condition (e.g. WHERE active = true)
UniqueEnforce uniqueness and speed lookups
Creating and using indexes (PostgreSQL)
-- single column
CREATE INDEX idx_trade_symbol ON trade(symbol);

-- composite: order matters. Good for WHERE symbol=? AND trade_date=?
CREATE INDEX idx_trade_sym_date ON trade(symbol, trade_date);

-- covering index: query is answered entirely from the index
CREATE INDEX idx_trade_cover ON trade(symbol) INCLUDE (qty, price);

-- partial index: only the rows you actually query
CREATE INDEX idx_active_cust ON customer(email) WHERE active = true;

-- see whether the planner uses it
EXPLAIN ANALYZE
SELECT qty, price FROM trade WHERE symbol = 'AAPL' AND trade_date = '2026-06-26';
-- Look for "Index Scan using idx_trade_sym_date" (good)
-- vs "Seq Scan on trade" (bad: full table scan)

The leftmost-prefix rule (top interview trap)

A composite index on (symbol, trade_date) can serve WHERE symbol = ? and WHERE symbol = ? AND trade_date = ?, but not WHERE trade_date = ? alone — because the index is sorted by symbol first. Order the columns by how you query, most-selective equality columns first.

Other "fast retrieval" levers (mention these to stand out)

  • Selectivity: index high-cardinality columns (many distinct values). Indexing a boolean rarely helps.
  • Covering indexes avoid the extra "heap fetch" back to the table.
  • Keep indexes usable: a function on the column (WHERE UPPER(symbol)=…) disables a plain index — use an expression index instead.
  • Partitioning for very large tables (by date/range) so scans hit one partition.
  • Statistics: run ANALYZE so the planner has fresh row estimates.
  • Pagination: prefer keyset pagination (WHERE id > ?) over large OFFSET.
Gotchas

Indexes are not free: every INSERT/UPDATE/DELETE must maintain them, so over-indexing slows writes and bloats storage — a real concern for high-throughput ETL targets. A query can usually use only one index per table (Postgres can combine via bitmap, but plan for one). Leading wildcards (LIKE '%abc') and column-side functions defeat B-trees. Always confirm with EXPLAIN ANALYZE rather than assuming.

Likely interview questions
What is an index and how does it speed up queries?

A sorted auxiliary structure (B-tree) on one or more columns. It turns a full O(n) scan into an O(log n) descent to the matching rows, at the cost of extra storage and slower writes.

Why might the database ignore your index?

Low selectivity (a scan is cheaper), stale statistics, a function/implicit cast on the indexed column, a leading-wildcard LIKE, or the query violating the leftmost-prefix rule of a composite index.

Clustered vs non-clustered index?

A clustered index defines the physical row order (one per table — e.g. SQL Server's primary key, or Postgres' CLUSTER). Non-clustered indexes are separate structures pointing back to the rows. In Postgres the table is a heap and all indexes are "secondary".

How would you diagnose a slow query?

Run EXPLAIN ANALYZE, look for seq scans on big tables, check row-estimate vs actual (stats drift), confirm the right index exists and is leftmost-usable, consider a covering or partial index, and re-ANALYZE.

Hands-on

In the Postgres you'll run for P6, create a trade table, insert ~500k rows with generate_series, then time SELECT … WHERE symbol='AAPL' before and after CREATE INDEX, comparing EXPLAIN ANALYZE output. Seeing "Seq Scan" flip to "Index Scan" with a 100×+ speedup is the most convincing thing you can describe in an interview.

Phase 1 · 1 weekJava 8the big jump

Java 8 Core — Lambdas, Streams, Optional, java.time

Java 8 (2014) is the watershed between "old" and "modern" Java. If you internalise Streams, most modern code becomes readable. Functional interfaces (A4) are the engine; Streams are the assembly line built on them.

Streams API — the pipeline you already think in

A Stream is a declarative pipeline over a collection: a source, zero or more lazy intermediate operations (each returns a new stream), and one terminal operation that triggers execution and produces a result. Nothing runs until the terminal op.

ETL analogy

This is almost literally a Pentaho/Informatica mapping: stream() = the reader, filter() = a filter step, map() = an expression/transform, sorted()/distinct() = sort/dedup, collect()/toList() = the writer. Laziness = the pipeline only does work when the target pulls rows.

From loop to stream (the slide your client already has)
// Java 7 — imperative
List<Employee> result = new ArrayList<>();
for (Employee e : employees) {
    if (e.getSalary() > 100000) result.add(e);
}

// Java 8+ — declarative
List<Employee> rich = employees.stream()
        .filter(e -> e.getSalary() > 100000)
        .toList();
The operations you'll use 90% of the time
import java.util.stream.*;

// map + filter + collect
List<String> names = employees.stream()
        .filter(e -> e.getSalary() > 100000)
        .map(Employee::getName)
        .sorted()
        .toList();

// grouping (like GROUP BY)
Map<String, List<Employee>> byDept = employees.stream()
        .collect(Collectors.groupingBy(Employee::getDept));

// aggregate per group (GROUP BY dept, AVG(salary))
Map<String, Double> avgByDept = employees.stream()
        .collect(Collectors.groupingBy(Employee::getDept,
                 Collectors.averagingDouble(Employee::getSalary)));

// reduce to a single value
double total = employees.stream()
        .mapToDouble(Employee::getSalary)   // primitive stream avoids boxing
        .sum();

// counting + partitioning
Map<Boolean, Long> split = employees.stream()
        .collect(Collectors.partitioningBy(e -> e.getSalary() > 100000,
                 Collectors.counting()));

Optional — no more NullPointerException reflexes

Optional<T> is a box that holds a value or nothing, forcing the caller to handle "absent" explicitly instead of returning null.

Optional<Employee> top = employees.stream()
        .max(Comparator.comparingDouble(Employee::getSalary));

String name = top.map(Employee::getName)      // transform if present
                 .orElse("none");             // fallback if absent

top.ifPresent(e -> System.out.println(e.getName()));   // act only if present

Method references — shorthand for obvious lambdas

employees.stream().map(Employee::getName)      // e -> e.getName()
                  .forEach(System.out::println);  // s -> System.out.println(s)
List<Employee> copy = employees.stream().sorted(Comparator.comparing(Employee::getName)).toList();

java.time — the modern date/time API

Replaces the broken, mutable java.util.Date/Calendar. The types are immutable and clearly named.

import java.time.*;

LocalDate today   = LocalDate.now();
LocalDate settle  = today.plusDays(2);                 // T+2
LocalDateTime ts  = LocalDateTime.now();
Duration d        = Duration.between(ts, ts.plusHours(3));
Period age        = Period.between(LocalDate.of(1990,1,1), today);
boolean overdue   = settle.isBefore(today);
Gotchas

A stream is single-use — consume it once; reusing throws IllegalStateException. Streams don't mutate the source, they produce results. Don't do I/O or mutate shared state inside map()/filter() (no side effects). Use primitive streams (mapToInt/Double) for number crunching to avoid boxing. Don't reach for parallelStream() unless you've measured — it's rarely worth it and easy to misuse.

Likely interview questions
Intermediate vs terminal operations?

Intermediate ops (filter, map, sorted) are lazy and return a new stream; terminal ops (collect, forEach, reduce, count) trigger execution and produce a result or side effect.

map() vs flatMap()?

map is 1→1 (transform each element). flatMap is 1→many flattened into one stream — e.g. turning List<Order> with line items into a single stream of all line items.

Why Optional instead of returning null?

It makes "no value" part of the type, so callers must handle absence; it composes with map/filter/orElse and reduces NPEs. Use it for return types, not fields or method parameters.

What is a Collector?

A recipe for accumulating stream elements into a result — toList(), toMap(), groupingBy(), joining(), counting(). It's the "writer/aggregator" end of the pipeline.

Hands-on

Take your BatchJobRun list from A1. Using Streams: (1) total rows written across all runs, (2) group runs by job name, (3) the run with the most skipped rows as an Optional, (4) a comma-joined string of job names sorted alphabetically (Collectors.joining(", ")). Rewrite each as the old for-loop too, to feel the difference.

Phase 2 · 2 weeksJava 9–17

Java 9–17 — var, Records, Text Blocks, Switch Expressions, Sealed Classes

These features remove boilerplate and make data modelling safer. Each is small on its own; together they're "modern Java style."

var — local type inference (Java 10)

The compiler infers the type from the right-hand side. It's still statically typed — only the typing is shorter. Use it when the type is obvious; spell the type out when it aids readability.

var name = "Nag";                       // String
var totals = new HashMap<String,Integer>();   // HashMap<String,Integer>
for (var e : employees) { /* ... */ }   // Employee inferred
// var x;          // ERROR: needs an initialiser
// var y = null;   // ERROR: nothing to infer

Records — immutable data carriers (Java 16)

A record auto-generates the constructor, private final fields, accessors, and correct equals()/hashCode()/toString(). This is the single biggest boilerplate killer — perfect for DTOs, API responses, and map keys (recall A3).

public record Employee(int id, String name, double salary) {}

// equivalent to ~40 lines of a classic POJO. Usage:
var e = new Employee(1, "Asha", 120000);
e.name();                 // accessor (no "get" prefix)
e.equals(new Employee(1,"Asha",120000));  // true — value equality for free

// compact constructor for validation
public record Money(String ccy, long minorUnits) {
    public Money {                       // runs before fields are assigned
        if (minorUnits < 0) throw new IllegalArgumentException("negative");
    }
}

Text blocks — multi-line strings (Java 15)

Triple-quoted strings for SQL, JSON, HTML — no \n or escaping noise. Indentation is stripped relative to the closing delimiter.

String sql = """
        SELECT id, name, salary
        FROM   employee
        WHERE  salary > ?
        ORDER  BY salary DESC
        """;

String json = """
        { "symbol": "AAPL", "qty": %d }
        """.formatted(500);

Switch expressions — switch that returns a value (Java 14)

Arrow syntax, no fall-through, no break, and it can be assigned. yield returns from a multi-line branch.

String label = switch (status) {
    case "A" -> "Active";
    case "I" -> "Inactive";
    case "P", "H" -> "Pending";          // multiple labels
    default -> "Unknown";
};

int score = switch (grade) {
    case "A" -> { int base = 90; yield base + 5; }   // block form
    default  -> 0;
};

Sealed classes — control who can extend (Java 17)

A sealed type lists exactly which classes may implement/extend it. This makes your domain model exhaustive: combined with switch pattern matching (P3/A6), the compiler guarantees you've handled every case.

public sealed interface PaymentResult
        permits Approved, Declined, Pending {}

public record Approved(String authCode) implements PaymentResult {}
public record Declined(String reason)   implements PaymentResult {}
public record Pending(long retryAfter)  implements PaymentResult {}
// No other class can implement PaymentResult — the set is closed.
ETL analogy

Records are your row/DTO definitions with zero ceremony — think of a strongly-typed record schema. Sealed types are like a constrained domain/reference list: only these statuses are legal, and downstream logic must handle each one. Text blocks are for the SQL you embed everywhere.

Gotchas

Records are immutable — no setters; "change" means create a new record. Records can't extend a class (they implicitly extend Record) but can implement interfaces. Don't overuse var where it hides an unclear type. A sealed type's permitted subclasses must be in the same module/package and must themselves be final, sealed, or non-sealed.

Likely interview questions
What does a record give you for free?

A canonical constructor, private final fields, public accessors, and value-based equals(), hashCode(), toString(). Ideal for immutable DTOs and map keys.

Switch statement vs switch expression?

The expression form (->) returns a value, has no fall-through, requires exhaustiveness, and avoids accidental break bugs. The classic statement form is still available for side-effecting code.

Why seal a class hierarchy?

To close the set of subtypes so the compiler can verify exhaustive handling (great with pattern-matching switch) and so the domain can't be extended in unexpected ways — safer modelling of a finite set of cases.

Is var dynamic typing?

No. The type is fixed at compile time by inference; var x = 1 is an int forever. It's purely a syntax convenience for locals.

Hands-on

Re-model your Employee as a record with a compact constructor rejecting negative salary. Add a sealed interface Status permits Active, Inactive, OnLeave and a switch expression turning a Status into a display string. Store an SQL query in a text block. You've now used four Phase-2 features in one tiny file.

Phase 3 · 1 weekJava 17–21virtual threads = huge

Java 17–21 — Virtual Threads, Pattern Matching, Structured Concurrency

Virtual Threads (Java 21) — the headline

A platform thread maps 1:1 to an OS thread and costs ~1 MB of stack, so a server can run maybe a few thousand. A virtual thread is managed by the JVM and costs almost nothing; you can run millions. When a virtual thread blocks on I/O, the JVM unmounts it from its carrier OS thread and runs another — so simple blocking code scales like hand-written async, without the callback spaghetti.

Virtual threads in practice
import java.util.concurrent.*;

// Run 10,000 blocking calls concurrently — no pool tuning
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    List<Future<String>> futures = new ArrayList<>();
    for (int i = 0; i < 10_000; i++) {
        int id = i;
        futures.add(executor.submit(() -> fetchQuote(id)));  // blocking I/O is fine
    }
    for (var f : futures) System.out.println(f.get());
}   // executor.close() waits for all tasks

// Or a one-off
Thread.startVirtualThread(() -> System.out.println("on a virtual thread"));

In Spring Boot 3.2+ you flip them on for the whole web layer with one property: spring.threads.virtual.enabled=true.

Pattern Matching (Java 16 instanceof → Java 21 switch + records)

// instanceof pattern — bind and test in one step (Java 16)
if (obj instanceof String s && s.length() > 3) {
    System.out.println(s.toUpperCase());      // 's' already typed as String
}

// switch + record patterns + guards (Java 21)
sealed interface Shape permits Circle, Rect {}
record Circle(double r) implements Shape {}
record Rect(double w, double h) implements Shape {}

double area(Shape sh) {
    return switch (sh) {
        case Circle(double r)            -> Math.PI * r * r;
        case Rect(double w, double h) when w == h -> w * w;   // guarded
        case Rect(double w, double h)    -> w * h;
    };   // exhaustive — sealed type means no default needed
}

Structured Concurrency (preview in Java 21 — understand conceptually)

It treats several related subtasks as a single unit: if one fails, the others are cancelled; the parent waits for all. It turns "fan out, then join" into something with clear ownership and error propagation, instead of leaking threads on failure.

// PREVIEW in 21: compile/run with --enable-preview
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    var user  = scope.fork(() -> loadUser(id));      // subtask 1
    var orders = scope.fork(() -> loadOrders(id));   // subtask 2
    scope.join().throwIfFailed();                    // wait; propagate first failure
    return new Dashboard(user.get(), orders.get());
}   // if either fails, the other is cancelled automatically
ETL analogy

Virtual threads = massively parallel source reads / API lookups without exhausting a thread pool — scaling pipeline parallelism without scaling hardware. Structured concurrency = a parent job that fans out parallel sub-extracts and fails the whole job cleanly if any branch fails (no orphaned threads).

Gotchas

Virtual threads help I/O-bound work, not CPU-bound number crunching (that's still limited by cores). Avoid synchronized blocks around blocking calls — they can "pin" a virtual thread to its carrier (use ReentrantLock instead). Don't pool virtual threads — create one per task. Structured concurrency is preview in 21: know it, but say so.

Likely interview questions
Platform thread vs virtual thread?

Platform threads wrap OS threads (heavy, few thousand max). Virtual threads are JVM-scheduled, extremely cheap (millions), and unmount from a carrier OS thread when they block — so blocking code scales without async complexity.

When do virtual threads NOT help?

CPU-bound workloads — you're still bounded by cores. They shine for high-concurrency I/O (HTTP calls, DB queries, file reads).

What is pattern matching for switch?

Switching on an object's type (and destructured record components, with optional when guards) instead of a constant, binding variables in each branch — eliminating cast boilerplate and enabling exhaustive handling over sealed types.

What problem does structured concurrency solve?

Unmanaged executor tasks can leak on failure and have unclear lifecycles. Structured concurrency scopes related subtasks so failure cancels siblings and the parent reliably joins — clearer error handling and no orphaned threads.

Hands-on

Write a method that "fetches" 1,000 fake quotes (each Thread.sleep(100) to simulate I/O). Run it once with a fixed Executors.newFixedThreadPool(10) and once with newVirtualThreadPerTaskExecutor(); time both. The virtual-thread version finishing in a fraction of the time is the demo to describe in your interview.

Spring Boot — REST, DI, JPA, Errors, Validation, Security

Spring Boot lets you build a production web service with almost no configuration: an embedded Tomcat, sensible defaults ("auto-configuration"), and "starter" dependencies that pull in everything a feature needs. You write business code; Boot wires the plumbing.

Version note for your interview

This guide targets Spring Boot 3.5.x on JDK 21 — what most enterprises run and what interview questions assume. As of mid-2026 the newest line is Spring Boot 4.x (built on Spring Framework 7, Java 17+ min, JSpecify null-safety, modularised jars); 3.5 is the final 3.x release and its open-source support ends 30 June 2026. Everything below is API-stable across 3.x; on 4.x the main differences you'd mention are Security 7 config changes and the Framework-7 baseline. Knowing that 3.x→4.x story is itself a good talking point.

The layered architecture (memorise this shape)

1 · Dependency Injection (DI) & IoC — the heart of Spring

Instead of objects creating their own collaborators (new CustomerService()), the Spring container creates them ("beans") and injects them where needed. You declare what you need; Spring provides it. This is Inversion of Control: the framework owns object creation and lifetime.

@Service                                   // a managed bean
public class CustomerService {
    private final CustomerRepository repo;
    // constructor injection (preferred): Spring passes the repo bean in
    public CustomerService(CustomerRepository repo) { this.repo = repo; }
    public Customer findById(Long id) { /* ... */ }
}

Stereotype annotations mark beans: @Component (generic), @Service (business), @Repository (data, adds exception translation), @RestController (web). Prefer constructor injection (shown) over field injection — it makes dependencies explicit, final, and testable.

ETL analogy

DI is like a job scheduler that hands each transform its configured connections and resources at runtime, rather than each step hard-coding its own DB connection. You declare the dependency; the container supplies the wired-up instance.

2 · REST APIs

You saw the controller in A5. The flow: @RestController maps HTTP to methods, Jackson auto-converts JSON ↔ your records, status codes communicate outcome.

@RestController
@RequestMapping("/api/customers")
public class CustomerController {
    private final CustomerService service;
    public CustomerController(CustomerService service) { this.service = service; }

    @GetMapping("/{id}")
    public CustomerDto get(@PathVariable Long id) { return service.findById(id); }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public CustomerDto create(@RequestBody @Valid CreateCustomer body) {
        return service.create(body);
    }
}

public record CreateCustomer(@NotBlank String name, @Email String email) {}
public record CustomerDto(Long id, String name, String email) {}

3 · Spring Data JPA — persistence without SQL boilerplate

JPA maps Java objects to table rows (ORM). Spring Data goes further: you declare a repository interface and Spring generates the implementation, including queries derived from method names.

@Entity
@Table(name = "customer")
public class Customer {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;
    private String email;
    private boolean active;
    // getters/setters (or use Lombok) ...
}

public interface CustomerRepository extends JpaRepository<Customer, Long> {
    // CRUD (save, findById, findAll, deleteById) is inherited — zero code.
    List<Customer> findByActiveTrue();                 // derived query
    Optional<Customer> findByEmail(String email);      // derived query

    @Query("select c from Customer c where c.name like %:q%")
    List<Customer> search(@Param("q") String q);       // explicit JPQL
}
ETL analogy

The repository is a generated data-access layer — like a reusable target/lookup connector. Method-name queries (findByActiveTrue) are the framework writing the SQL for you; @Query is when you take the wheel, exactly like dropping to hand-written SQL in a mapping.

4 · Exception Handling — one place, consistent errors

Throw meaningful exceptions in your service; translate them centrally so every error returns the right status and a uniform body. This is the @ControllerAdvice in your notes (its REST-specific form is @RestControllerAdvice).

public class NotFoundException extends RuntimeException {
    public NotFoundException(String m) { super(m); }
}

@RestControllerAdvice
public class ApiExceptionHandler {

    @ExceptionHandler(NotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)                 // 404
    public ApiError handleNotFound(NotFoundException ex) {
        return new ApiError("NOT_FOUND", ex.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)               // 400
    public ApiError handleValidation(MethodArgumentNotValidException ex) {
        String msg = ex.getBindingResult().getFieldErrors().stream()
                .map(f -> f.getField() + ": " + f.getDefaultMessage())
                .collect(Collectors.joining(", "));
        return new ApiError("VALIDATION", msg);
    }
}
public record ApiError(String code, String message) {}

5 · Validation — reject bad input at the edge

Bean Validation annotations on your request record + @Valid on the parameter make Spring validate before your method runs; failures become a 400 via the handler above. Needs the spring-boot-starter-validation starter.

public record CreateCustomer(
        @NotBlank String name,
        @Email String email,
        @Min(18) int age,
        @Size(max = 500) String notes
) {}
// @NotNull @NotBlank @Email @Min @Max @Size @Pattern @Positive ...

6 · Security Basics — JWT, OAuth2, Keycloak

Spring Security intercepts every request through a filter chain and decides authentication (who are you?) and authorisation (what may you do?). The modern stateless pattern for APIs: the client sends a JWT (a signed token) on each request; the server validates the signature and reads the claims — no server-side session (which fits REST's statelessness, A5).

  • JWT — a compact signed token (header.payload.signature) carrying claims (user, roles, expiry). Self-contained: the server trusts it because the signature checks out.
  • OAuth2 / OIDC — the protocol for obtaining tokens from an authorisation server, so your app never handles passwords directly.
  • Keycloak — a popular open-source identity provider that issues those OAuth2/OIDC tokens (users, roles, login UI) so you don't build auth yourself.
// Minimal stateless resource-server config (Spring Security 6, Boot 3.x)
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Bean
    SecurityFilterChain chain(HttpSecurity http) throws Exception {
        http.csrf(csrf -> csrf.disable())
            .authorizeHttpRequests(reg -> reg
                .requestMatchers("/api/public/**").permitAll()
                .anyRequest().authenticated())
            .oauth2ResourceServer(o -> o.jwt(Customizer.withDefaults())); // validate JWTs
        return http.build();
    }
}

For learning, you don't need Keycloak on day one — Spring Security's HTTP Basic or an in-memory user gets you started; add JWT/Keycloak once the basics click.

Gotchas

Don't expose entities directly — map to DTO records (avoids lazy-loading serialization errors and leaking columns). N+1 queries: lazy associations in a loop fire one query per row; fix with JOIN FETCH or an @EntityGraph. Put @Transactional on the service, not the controller. In Security 6/Boot 3 the old WebSecurityConfigurerAdapter is gone — configure a SecurityFilterChain bean as above. Never log or commit secrets; never disable CSRF on a browser-session app (it's fine for a stateless token API).

Likely interview questions
What is dependency injection and why use it?

The container creates and supplies an object's collaborators instead of the object building them. Benefits: loose coupling, easy swapping/mocking for tests, single place to manage lifecycle and config. Prefer constructor injection.

What does Spring Boot auto-configuration do?

Based on the classpath and properties, it configures sensible beans automatically (datasource, web server, Jackson, etc.), which you can override. "Starters" bundle the dependencies for a feature.

How does Spring Data JPA generate queries?

From method-name conventions (findByEmailAndActiveTrue) it parses into a query; for complex cases you supply @Query JPQL/native SQL. CRUD comes from JpaRepository.

How do you centralise error handling?

@RestControllerAdvice with @ExceptionHandler methods that map exceptions to HTTP status + a consistent error body, keeping controllers clean.

Session vs JWT authentication?

Sessions store state server-side (a cookie holds the session id). JWTs are stateless: the signed token carries the claims, so any server instance can validate it without shared session storage — better for scaling and REST.

Spring Batch — Your ETL Mental Model, One-to-One

This is the phase where your 20 years pay off immediately. Spring Batch is an ETL framework: it reads items, processes them, and writes them in chunks, with restartability, transaction boundaries, skip/retry, and run history — the same concerns you've handled in Informatica and Pentaho. You already know the concepts; you're just learning the Java names.

The mapping (put this on a sticky note)

Chunk-oriented processing (the core idea)

Batch reads items one by one, but writes them in chunks of N inside a single transaction. Read N, process N, write N, commit — repeat. If the write fails, that chunk's transaction rolls back; the framework records progress so you can restart from where it stopped. This is exactly "commit interval" tuning from your ETL world.

A CSV → transform → DB step (Spring Batch 5 API)
@Configuration
public class BatchConfig {

    // SOURCE: read a CSV line into a Customer (FlatFileItemReader)
    @Bean
    public FlatFileItemReader<Customer> reader() {
        return new FlatFileItemReaderBuilder<Customer>()
                .name("customerReader")
                .resource(new ClassPathResource("customers.csv"))
                .delimited().names("name", "email", "active")
                .targetType(Customer.class)
                .linesToSkip(1)                 // header row
                .build();
    }

    // TRANSFORM: normalise; return null to FILTER an item out
    @Bean
    public ItemProcessor<Customer, Customer> processor() {
        return c -> {
            if (c.getEmail() == null || c.getEmail().isBlank()) return null; // drop
            c.setEmail(c.getEmail().toLowerCase());
            return c;
        };
    }

    // TARGET: insert each chunk via JPA
    @Bean
    public JpaItemWriter<Customer> writer(EntityManagerFactory emf) {
        var w = new JpaItemWriter<Customer>();
        w.setEntityManagerFactory(emf);
        return w;
    }

    // STEP: wire R→P→W with a chunk size (commit interval)
    @Bean
    public Step importStep(JobRepository jobRepository,
                           PlatformTransactionManager txManager,
                           FlatFileItemReader<Customer> reader,
                           ItemProcessor<Customer, Customer> processor,
                           JpaItemWriter<Customer> writer) {
        return new StepBuilder("importStep", jobRepository)
                .<Customer, Customer>chunk(100, txManager)   // 100 = commit interval
                .reader(reader).processor(processor).writer(writer)
                .faultTolerant().skipLimit(10).skip(FlatFileParseException.class)
                .build();
    }

    // JOB: an ordered set of steps
    @Bean
    public Job importJob(JobRepository jobRepository, Step importStep) {
        return new JobBuilder("importJob", jobRepository)
                .start(importStep)
                .build();
    }
}
ETL analogy

Read the code above as a mapping: FlatFileItemReader is your flat-file source, the ItemProcessor lambda is an expression/filter transform (returning null = a router dropping the row), JpaItemWriter is the target, chunk(100) is your commit interval, and faultTolerant().skipLimit() is your reject/error-handling strategy. The metadata tables Spring Batch maintains are your run/session logs.

Gotchas

Spring Batch needs its metadata tables (BATCH_JOB_*); Boot can auto-create them (spring.batch.jdbc.initialize-schema=always for dev). A JobInstance is identified by job name + identifying JobParameters — re-running with the same parameters won't run a completed instance again (add a unique param like a run timestamp to force a new run). In Spring Batch 5 the old JobBuilderFactory/StepBuilderFactory are gone — build with JobBuilder/StepBuilder taking the JobRepository (as above). Chunk size is a tuning knob: too small = slow (many commits), too large = big transactions/memory.

Likely interview questions
Explain Job, Step, and chunk processing.

A Job is the whole batch process, made of ordered Steps. A chunk-oriented Step reads items via an ItemReader, transforms via an ItemProcessor, and writes N at a time via an ItemWriter, committing one transaction per chunk.

How does Spring Batch support restartability?

It persists execution state in metadata tables. On restart it skips completed steps and, for chunk steps, resumes from the last committed position, so you don't reprocess everything after a failure.

How do you skip/retry bad records?

Make the step faultTolerant() and declare skip(Exception.class) with a skipLimit, or retry(...) with a retryLimit for transient failures — the framework isolates the offending item.

How is this different from your Informatica/Pentaho work?

Conceptually identical — source/transform/target, commit intervals, reject handling, run logs. The difference is it's code-first and embeddable in a Spring Boot app/REST service rather than a separate GUI tool, which is exactly the migration story in Phase 6.

Hands-on

Add Spring Batch to your Phase-4 project. Drop a customers.csv on the classpath and build the Job above to load it into Postgres, with the lowercase-email transform and a skip for malformed lines. Run it, then query the BATCH_JOB_EXECUTION table to see your run history — that's your "session log," now self-managed.

  batch:
    jdbc.initialize-schema: always  # create BATCH_* metadata tables (dev)
    job.enabled: false              # don't auto-run on startup; trigger via REST

Trigger the job from REST (the modern touch)

@RestController
@RequestMapping("/api/jobs")
public class JobController {
    private final JobLauncher launcher;
    private final Job importJob;
    public JobController(JobLauncher launcher, Job importJob) {
        this.launcher = launcher; this.importJob = importJob;
    }

    @PostMapping("/import")
    public Map<String,Object> run() throws Exception {
        var params = new JobParametersBuilder()
                .addLong("ts", System.currentTimeMillis())   // unique → fresh run
                .toJobParameters();
        var exec = launcher.run(importJob, params);
        return Map.of("status", exec.getStatus().toString(),
                      "id", exec.getId());
    }
}

ETL analogy

You're not learning a new domain — you're re-platforming a familiar one. The CSV source, the lowercase/validation transform, the Postgres target, the commit interval, the reject handling: all concepts you've shipped for years. The only new thing is expressing them as Spring Batch beans inside a deployable service.

Gotchas

Set spring.batch.job.enabled=false so the job doesn't fire on every startup — you want to trigger it via REST. Add a unique JobParameter per run or a completed instance won't re-run. ddl-auto: update and initialize-schema: always are fine for a demo but say "I'd use Flyway/Liquibase migrations in production" — interviewers listen for that. Keep secrets out of application.yml (use env vars); don't commit them.

Likely interview questions
Walk me through migrating an Informatica mapping to Spring Batch.

Map source→ItemReader, transformations→ItemProcessor, target→ItemWriter, the mapping→a Step, the workflow→a Job; set the chunk size from the old commit interval, port reject handling to skip/retry, and replace the scheduler/session log with Spring Batch's metadata tables — optionally triggering via REST instead of a scheduler.

Why move off a GUI ETL tool to code?

Version control and code review, unit/integration testing, CI/CD, no licence cost, easier embedding in microservices, and on-demand triggering. Trade-off: you lose the visual designer and some out-of-the-box connectors, so it's a case-by-case call.

How would you scale this for large volumes?

Tune chunk size, use partitioned or multi-threaded steps, switch to a high-throughput writer (JdbcBatchItemWriter with batched inserts), index the target appropriately (A7), and consider parallel steps for independent feeds.

Either extension gives you a concrete, confident answer to "tell me about a recent project."
